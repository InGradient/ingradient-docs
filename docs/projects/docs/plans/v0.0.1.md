## Feature 2: Gateway + Auth + Job Lifecycle

### Phase 2 — API 설계

**Job API** (`docs/api_contract.md` 기준):

| Method | Path | 요청 핵심 필드 | 응답 | 에러 |
|--------|------|--------------|------|------|
| `POST` | `/api/jobs` | job_type, task_type, runtime, model_id, recipe_name, resource_request, idempotency_key | Job | validation_error, unauthorized, conflict |
| `GET` | `/api/jobs` | ?status, runtime, model, queue, worker, priority, backend, from, to, limit, cursor | Job[] + pagination | - |
| `GET` | `/api/jobs/:id` | - | Job + latest_attempt | not_found |
| `POST` | `/api/jobs/:id/cancel` | - | Job | invalid_state_transition |
| `POST` | `/api/jobs/:id/retry` | - | Job | invalid_state_transition |
| `POST` | `/api/jobs/:id/clone` | overrides (optional) | Job | - |
| `POST` | `/api/jobs/:id/reprioritize` | { priority } | Job | - |

**응답 envelope:**
```json
{ "data": {}, "meta": { "request_id": "...", "api_version": "v1" } }
{ "data": [], "meta": { ..., "pagination": { "next_cursor": "...", "has_more": true, "total_count": 50 } } }
{ "error": { "code": "validation_error", "message": "...", "details": {}, "retryable": false }, "meta": {} }
```

**Auth 흐름:**
1. 요청 header: `Authorization: Bearer {token}`
2. Gateway → auth-service `POST /verify` → `{ valid, role, scope[] }`
3. admin role 필수 (모든 endpoint)
4. service-to-service: `X-Service-Token` 헤더
5. 실패: `401 unauthorized` / `403 forbidden`

**Idempotency key:**
- `X-Idempotency-Key` 헤더 또는 body `idempotency_key` 필드
- 24시간 유효, 동일 key → 기존 job 반환 (새 job 생성 안 함)
- 저장: `jobs.idempotency_key` 컬럼 unique index

### Phase 3 — Job Lifecycle 서비스 로직

**상태 전이 허용 테이블:**

| 현재 상태 | 허용 전이 | 조건 |
|-----------|-----------|------|
| created | → queued | orchestrator가 검증 완료 후 |
| queued | → scheduled, → cancelled | scheduler 선택 / cancel 요청 |
| scheduled | → preparing | worker 배정 |
| preparing | → running | worker 준비 완료 |
| running | → succeeded, → failed, → cancel_requested, → timed_out | 결과 수신 / cancel signal / timeout |
| cancel_requested | → cancelled | worker graceful shutdown 완료 |
| failed | → queued (retry), → dead_letter | retry 정책 판단 |

**retry 정책** (`retry_policy_engine.md`):
- failure_code가 retryable 목록에 있으면 retry
- 최대 retry 횟수 초과 → dead_letter
- OOM: 즉시 재시도 금지 (동일 조건 방지)
- backoff: 지수 backoff (TBD)

**JobAttempt 생성 시점:** retry 시 attempt_no +1, 새 attempt record 생성

**이벤트 발행:** 상태 전이마다 → SSE로 전달 (F9 연동)

### Phase 3 — 테스트 케이스

- `POST /api/jobs` — 성공, 필수 필드 누락(400), 미지원 runtime(400), 중복 idempotency_key(200 기존 반환)
- `POST /api/jobs/:id/cancel` — queued → cancelled, running → cancel_requested, already cancelled(409)
- `POST /api/jobs/:id/retry` — failed → queued, non-failed 상태(409)
- 인증 없음 → 401, 비 admin → 403

### Phase 2 — API 설계

| Method | Path | 설명 |
|--------|------|------|
| `GET` | `/api/queues` | queue 목록 (runtime, backend 필터) |

| `POST` | `/api/queues/:name/pause` | - |

| `GET` | `/api/backends` | backend 목록 + 상태 |
| `GET` | `/api/backends/:id` | backend 상세 |

### Phase 3 — 라우팅 서비스 로직

**3단계 라우팅** (`execution_routing.md`):
1. backend 후보 필터링: `allow_external_execution`, `execution_preference`, `latency_class`
2. 후보 중 최적 선택: health/pressure/quota/preferred_backend 기준
3. fallback 정책: training pool 포화 → vertex_custom_job, inference wait > X초 → vertex_batch_prediction, GPU 없음 + allow_cpu_fallback → gke_cpu

**routing_reason_code 목록:** `preferred`, `least_loaded`, `cpu_fallback`, `overflow_vertex`, `only_available`

**overflow 조건 평가:**
- queue wait time > threshold → action 결정 (reject / offload / queue)
- queue별 threshold: realtime=15s, interactive=120s, batch=1800s (`training_scheduler.md` 기준)

## Feature 4: Training Scheduler (VRAM + GPU Sharing + OOM)

### Phase 3 — 스케줄러 로직

**GPU 후보 선택 순서** (`training_scheduler.md`):
1. runtime compatibility (worker.runtime == job.runtime)
2. worker health (healthy or idle만)
3. queue policy compatibility
4. GPU class compatibility
5. available_for_scheduling_mb ≥ job.resource_request.vram_class
6. sharing/exclusive 정책 (training → exclusive 필수)
7. model locality (로드된 모델 우선)
8. 최근 OOM 이력 (oom_recent worker 후순위)
9. node imbalance 완화

**VRAM 계산:**
```
available_for_scheduling_mb =
  total_vram_mb
  - system_reserve_mb (약 1GB)
  - runtime_reserve_mb (약 512MB)
  - fragmentation_margin_mb (약 512MB)
  - warm_footprint_mb (로드된 모델 합계)
  - active_job_footprint_mb (실행 중 job committed_vram 합계)
  - spike_buffer_mb (10% 여유)
```

**OOM 대응:**
- attempt에 peak_vram_mb, failure_code='oom' 기록
- 동일 worker 반복 OOM(3회/30분) → worker status = 'oom_recent'
- oom_recent worker는 drain 또는 maintenance 권고 alert 생성

## Feature 6: Runtime Contract + Utility Runtime

### Phase 3 — 인터페이스 정의

```typescript
interface Runtime {
  can_accept(job: Job): boolean
  prepare_model(model_ref: string): Promise<'ready' | 'skipped'>
  prepare_inputs(artifact_refs: string[]): Promise<PreparedInputs>
  execute(job_context: JobContext): Promise<ExecutionResult>
  collect_metrics(run_context: RunContext): Promise<Metrics>
  persist_outputs(result: ExecutionResult): Promise<string[]>  // artifact_refs

```

**Utility runtime 구현 대상 (v1):**
- `preprocess`: 이미지 resize, normalization, format conversion
- `export`: COCO/YOLO format 변환
- `report`: metrics summary 리포트 생성

**실행 환경:** `utility-pool`, CPU-only, `execution_preference=cpu`

**heartbeat:** 5초 간격, `{ worker_id, status, active_job_id, loaded_models, gpu_metrics }`
