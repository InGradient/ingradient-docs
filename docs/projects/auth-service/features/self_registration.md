# 자가 회원가입

- 문서명: 자가 회원가입
- 목적: 관리자 초대 없이 사용자가 직접 가입할 수 있는 흐름을 제공한다.
- 적용 범위: auth-service

- Owner: Backend

- 관련 SSOT 문서: gap-analysis.md

## 1. 기능 개요

- 기능 이름: 자가 회원가입
- 한 줄 정의: `POST /auth/signup`으로 사용자가 직접 계정을 생성하고, 선택적 이메일 인증 후 활성화되는 가입 흐름을 제공한다.

## 2. 문제 정의

- 현재 사용자 생성은 관리자만 가능하여 셀프서비스 시나리오를 지원하지 못한다.
- SaaS 모델에서 자가 가입은 필수적인 온보딩 경로이다.

## 3. 목표

- 사용자 목표: 초대 없이 직접 계정을 생성하고 서비스를 이용한다.
- 운영 목표: 가입 후 이메일 인증, 약관 동의 등 필수 절차를 자동화한다.

## 4. 범위 / 비범위

- 포함: `POST /auth/signup` 엔드포인트, 이메일+비밀번호 기반 가입, 조직 설정에 따른 자동 가입 활성화/비활성화
- 비범위: 소셜 가입(OAuth는 별도 SSO 흐름), 전화번호 가입

## 5. 사용자 시나리오

### 기본 흐름 — 자가 가입

1. 참여자: 미인증 사용자
2. 선행 조건: 가입이 허용된 시스템 설정
3. 기본 흐름: `POST /auth/signup` (email, password, name) → 계정 생성 (상태: PENDING_VERIFICATION 또는 ACTIVE) → 이메일 인증 활성화 시 인증 메일 발송
4. 예외 흐름: 이미 존재하는 이메일 → 409, 비밀번호 정책 미충족 → 400
5. 로그 포인트: `user_signup`

## 6. UI / UX 방향

- 로그인 페이지에 "계정 만들기" 링크
- 가입 폼: 이름, 이메일, 비밀번호, 비밀번호 확인, 약관 동의

## 7. 데이터 / API / 권한 영향

- 새 API: `POST /auth/signup`
- 시스템 설정 확장: `signup_enabled` (boolean), `default_org_id` (선택)
- 가입 시 기본 역할 자동 부여: MEMBER
- rate limit: signup 엔드포인트 5회/분

## 8. 테스트 계획

- unit: 입력 검증, 비밀번호 해싱, 중복 이메일 체크
- integration: 가입 → 이메일 인증 → 로그인 전체 플로우
- e2e: 가입 비활성화 시 403 응답 확인

## 9. 릴리즈 고려사항

- release note: 자가 회원가입 기능 추가
- 기본값: signup_enabled=false (기존 동작 유지)

## 10. 오픈 이슈

- 가입 시 조직 자동 생성 vs 기본 조직 배정 정책
- CAPTCHA 도입 시점
