# API 버저닝

- 문서명: API 버저닝
- 목적: API URL에 버전 접두사를 도입하여 하위 호환성을 보장한다.
- 적용 범위: auth-service
- 상태: Draft
- Owner: Backend
- Reviewer: —
- 마지막 수정일: 2026-03-28
- 관련 SSOT 문서: gap-analysis.md

## 1. 기능 개요

- 기능 이름: API 버저닝
- 한 줄 정의: 모든 API 경로에 `/v1/` 접두사를 도입하고, 향후 breaking change 시 새 버전으로 분리한다.

## 2. 문제 정의

- 현재 API에 버전 정보가 없어 breaking change 시 모든 소비자가 동시에 수정해야 한다.
- 여러 버전을 병행 운영할 수 없어 점진적 마이그레이션이 불가능하다.

## 3. 목표

- 사용자 목표: API 소비자가 자신의 속도에 맞게 새 버전으로 마이그레이션한다.
- 운영 목표: 이전 버전을 일정 기간 유지하면서 새 버전을 출시한다.

## 4. 범위 / 비범위

- 포함: `/v1/` 접두사 도입, 기존 경로 호환 리다이렉트, 버전 deprecation 헤더
- 비범위: 헤더 기반 버저닝, 버전별 독립 코드베이스

## 5. 사용자 시나리오

### 기본 흐름 — 버전 접두사 사용

1. 참여자: API 소비자
2. 선행 조건: 없음
3. 기본 흐름: `GET /v1/users` → 정상 응답. `GET /users` → `/v1/users`로 리다이렉트 (과도기)
4. 예외 흐름: 지원 중단 버전 → `Sunset` 헤더 + `Deprecation` 헤더 포함
5. 로그 포인트: `api_version_redirect`

## 6. UI / UX 방향

- API 문서에 버전별 엔드포인트 명시
- 버전 변경 마이그레이션 가이드 제공

## 7. 데이터 / API / 권한 영향

- 라우팅 변경: 모든 경로에 `/v1` 접두사 추가
- 호환성: 기존 경로 → `/v1/` 리다이렉트 미들웨어 (6개월 유지)
- 응답 헤더: `API-Version: v1`, `Deprecation`, `Sunset`

## 8. 테스트 계획

- unit: 라우팅 미들웨어 버전 분기 확인
- integration: `/v1/` 경로 정상 응답, 기존 경로 리다이렉트 확인
- e2e: 소비자(platform, admin-ui) 전환 테스트

## 9. 릴리즈 고려사항

- release note: API 버저닝 도입 (`/v1/` 접두사)
- 기존 경로 호환: 6개월 리다이렉트 기간 후 제거 예정
