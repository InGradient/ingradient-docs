# Debug Reading Guide

- 문서명: Debug Reading Guide
- 목적: 사소한 오류가 아닌 큰 문제, 반복 수정해도 해결되지 않는 문제를 다방면으로 분석하고 해결하기 위한 가이드다.
- 적용 범위: INGRADIENT 전체 프로젝트
- 상태: Draft
- Owner: Dev / Ops
- 마지막 수정일: 2026-03-28

> 이 가이드를 따르기 전에 `common_guide.md`를 먼저 읽는다. 소통, 테스트, 코드 품질, 문서 원칙이 모든 작업에 공통 적용된다.

## 이 가이드를 사용하는 경우

- 사소한 오류(오타, 단순 로직 실수 등)는 이 가이드 없이 바로 수정하면 된다
- **이 가이드는 아래 상황에서 사용한다:**
  - 원인을 알 수 없는 문제
  - 반복 수정해도 해결되지 않는 문제
  - 여러 서비스에 걸친 복합적인 문제
  - 재현이 어려운 간헐적 문제
  - 운영 장애

## 0단계: 디버깅 요청 시 확인할 것

**중요: 마음대로 진행하지 않는다.** 애매하거나 판단이 필요한 부분이 있으면 반드시 질문한다. 원인이 불확실하면 추측으로 수정하지 말고 먼저 공유한다.

디버깅 요청을 받았을 때 아래 항목이 명시되지 않았으면 먼저 확인한다.

| 확인 항목 | 누락 시 질문 |
|-----------|-------------|
| 대상 프로젝트 | 어떤 프로젝트(서비스)에서 발생한 문제인가? |
| 증상 | 어떤 에러/동작이 발생하는가? 에러 메시지가 있는가? |
| 재현 조건 | 어떤 상황에서 발생하는가? 항상 재현되는가, 간헐적인가? |
| 영향 범위 | 특정 사용자/환경인가, 전체인가? |
| 긴급도 | 운영 장애인가, 개발 중 발견인가, 사용 중 발견인가? |
| 환경 | dev / staging / production 중 어디에서 발생하는가? |

## 1단계: 기존 사례 확인

수정에 들어가기 전에 같은 문제가 이미 기록되어 있는지 먼저 확인한다.

- `../ops/troubleshooting.md` — 반복되는 운영 문제 진단 가이드
- `../ops/debug_notes.md` — 구조적이거나 재현이 어려운 디버깅 사례
- `../ops/incident_response.md` — 운영 장애 대응 절차 (긴급 상황일 때)

## 2단계: 관련 문서 파악

문제가 발생한 영역의 설계와 동작을 이해한다.

| 확인 대상 | 읽을 문서 |
|-----------|-----------|
| 해당 기능의 기획 | `projects/{프로젝트}/features/{기능명}.md` |
| 내부 구조 | `projects/{프로젝트}/architecture.md` |
| 데이터 모델 | `projects/{프로젝트}/data_model.md` |
| API 계약 | `projects/{프로젝트}/api_contract.md` |
| 권한 흐름 | `../product/role_permission_flow.md` |
| 서비스 간 연동 | `../product/multi_product_integration.md` |

## 3단계: 원인 조사

### 조사 절차

1. 증상을 명확히 기록한다 (에러 메시지, 스택 트레이스, 재현 조건)
2. **디버깅 로그를 적극적으로 추가한다.** 의심되는 지점에 상세 로그를 남겨 흐름을 추적한다. 로그는 많을수록 좋다 — 원인 파악 후 정리한다
3. 관련 로그를 확인한다
   - 서버 로그, 클라이언트 로그, DB 쿼리 로그
   - `../ops/monitoring_observability.md` 참고
4. 최근 변경사항을 확인한다 (git log, 최근 배포)
5. 문제 범위를 좁힌다
   - 특정 엔드포인트? 특정 입력? 특정 상태?
   - 재현 가능한 최소 조건을 찾는다
6. **한 방향에서 막히면 다른 방향을 시도한다.** 같은 접근을 반복하지 않는다. 코드, 로그, 네트워크, DB, 환경변수, 타이밍 등 다방면에서 분석한다

### 서비스별 디버깅 포인트

| 서비스 | 주요 확인 포인트 |
|--------|-----------------|
| auth-service | 토큰 만료/갱신, 멤버십 상태, 계정 잠금, Entitlement 검증 |
| platform | 데이터셋/이미지 상태, 권한 매트릭스, Export job 상태, Training 상태 전이 |
| edge | sync 상태 (local_only/uploading/synced/upload_failed), 카메라 연결, 오프라인 진입/복귀 |
| ai | job 상태 전이, GPU 큐, worker 연결, artifact 생성 |

## 4단계: 수정

### 수정 원칙

- 원인이 확실해진 후에 수정한다. 추측으로 수정하지 않는다
- 수정 범위를 최소화한다. 버그 수정에 리팩토링을 섞지 않는다
- 수정 전에 버그를 재현하는 테스트를 먼저 작성한다
- 수정 후 해당 테스트가 통과하는지 확인한다

### 디버깅 로그 정리

수정이 완료되면 디버깅용으로 추가한 로그를 제거할지 확인한다.
- 디버깅 전용 로그 (임시 console.log, verbose 출력 등)는 제거 대상
- 운영에 도움이 되는 로그 (에러 추적, 상태 전이 등)는 유지
- **반드시 "디버깅 로그를 정리할까요?" 라고 확인한 뒤 진행한다**

### 회귀 테스트 작성

- 버그를 재현하는 테스트를 반드시 남긴다
- 테스트 작성은 `test_reading_guide.md` 3단계 참고

## 5단계: 결과 기록

해결 후 재발 가능성이 있으면 기록을 남긴다.

### 기록 기준

사소한 오류는 기록하지 않는다. 아래에 해당하는 경우만 기록한다.

| 조건 | 기록 위치 | 기록 형식 |
|------|-----------|-----------|
| 당장 해결했지만 사용자 입장에서 다시 발생할 수 있는 문제 | `../ops/troubleshooting.md` | 증상 → 가능한 원인 → 확인 순서 → 복구 방법 |
| 구조적이거나 재현이 어렵거나 해결에 오래 걸린 문제 | `../ops/debug_notes.md` | 증상 → 재현 조건 → 원인 → 해결 방식 → 재발 방지 |
| 운영 장애였던 경우 | `../ops/incident_response.md` | 장애 대응 절차에 따라 기록 |

### 문서 업데이트

| 문서 | 수정 내용 |
|------|-----------|
| `releases/v{버전}.md` | 버그 수정 항목 체크, 수정 내용 기록 |
| `features/{기능명}.md` | 오픈 이슈에서 해결 완료로 이동, 필요 시 시나리오 보완 |
