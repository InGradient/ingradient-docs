# Patch

$ARGUMENTS 수정을 진행한다. 기존 기능의 디자인이나 동작을 부분 변경한다.

**이 스킬은 아래 상황에서 사용한다:**
- UI 디자인 변경 (색상, 레이아웃, 컴포넌트 교체 등)
- 기존 API에 필드 추가/변경
- 기존 화면에 기능 추가 (필터, 정렬, 버튼 등)
- 텍스트/라벨/메시지 수정
- 동작 방식 변경 (확인 다이얼로그 추가, 순서 변경 등)

**사용하지 않는 경우:**
- 새 기능 구현 → /0-plan-feature
- 버그 수정 → /debug
- 전체 리뷰 → /review

## 0단계: 사전 확인

| 확인 항목 | 누락 시 질문 |
|-----------|-------------|
| 수정 대상 | 어떤 화면/기능/API를 수정하는가? |
| 변경 내용 | 구체적으로 무엇을 어떻게 바꾸는가? |
| 이유 | 왜 변경이 필요한가? (기획 변경, 사용성 개선, 피드백 반영 등) |
| 관련 문서 | 기획 문서나 와이어프레임도 함께 수정해야 하는가? |

## 1단계: 현재 상태 파악

수정 전에 현재 코드의 상태를 먼저 파악한다.

**읽을 문서:**

| 문서 | 목적 |
|------|------|
| 관련 features/ 문서 | 현재 기획된 동작 — 수정이 기획과 충돌하는지 확인 |
| 관련 wireframes/ 문서 | 현재 화면 구조 — UI 수정이면 반드시 확인 |
| api_contract.md | 현재 API 구조 — API 수정이면 반드시 확인 |
| data_model.md | 현재 DB 구조 — DB 수정이면 반드시 확인 |

**코드 파악:**

- [ ] 수정 대상 코드를 읽고 현재 동작을 이해한다
- [ ] 수정이 영향을 주는 다른 코드를 파악한다 (이 컴포넌트를 사용하는 곳, 이 API를 호출하는 곳)
- [ ] 기존 테스트가 있는지 확인한다

## 2단계: 영향 범위 분석

수정이 다른 곳에 영향을 주는지 분석한다. Plan에 직접 기록한다.

    ## 영향 범위
    - 수정 대상: [파일:라인]
    - 이 코드를 사용하는 곳: [파일 목록]
    - 기획 문서 변경 필요: [있으면 명시, 없으면 "없음"]
    - wireframe 변경 필요: [있으면 명시, 없으면 "없음"]
    - api_contract 변경 필요: [있으면 명시, 없으면 "없음"]
    - data_model 변경 필요: [있으면 명시, 없으면 "없음"]
    - 다른 서비스 영향: [있으면 명시, 없으면 "없음"]

영향 범위가 예상보다 크면 사용자에게 알리고 진행 여부를 확인한다.

## 3단계: Plan 작성

수정할 내용을 구체적으로 Plan에 작성한다.

**필수 포함 항목:**

- [ ] 변경 전 상태 (현재 코드가 어떻게 동작하는지)
- [ ] 변경 후 상태 (수정 후 어떻게 동작해야 하는지)
- [ ] 수정할 파일 목록
- [ ] 각 파일에서 변경할 구체적 내용
- [ ] 영향받는 테스트 — 기존 테스트 수정 또는 새 테스트 추가

## 4단계: 구현

**규칙:**

- **요청받은 범위만 정확히 변경한다.** 주변 코드를 "개선"하지 않는다
- **기존 코드 패턴을 따른다.** 수정하면서 새 패턴을 도입하지 않는다
- **내 변경으로 생긴 고아 코드만 정리한다.** 관련 없는 dead code를 건드리지 않는다
- **변경한 모든 줄이 요청과 직접 연결되는지 확인한다**

## 5단계: 테스트

- [ ] 기존 테스트가 깨지지 않는가
- [ ] 수정된 동작에 대한 테스트가 추가/수정되었는가
- [ ] UI 수정이면 Playwright 테스트가 업데이트되었는가

## 6단계: 문서 업데이트

수정 내용에 따라 문서를 업데이트한다.

| 조건 | 수정할 문서 |
|------|-------------|
| UI가 바뀌면 | wireframes/ 관련 문서 |
| 기능 동작이 바뀌면 | features/ 관련 문서 |
| API가 바뀌면 | api_contract.md |
| DB가 바뀌면 | data_model.md |

**문서를 수정해야 하는지 판단이 어려우면 질문한다.** 임의로 수정하지 않는다.

## 완료 기준

- [ ] 요청된 수정이 구현되었다
- [ ] 요청 범위 외의 코드를 건드리지 않았다
- [ ] 기존 테스트가 통과한다
- [ ] 수정에 대한 테스트가 추가/수정되었다
- [ ] 영향받는 문서가 업데이트되었다
