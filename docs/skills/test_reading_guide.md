# Test Reading Guide

- 문서명: Test Reading Guide
- 목적: 테스트를 수행할 때 읽어야 하는 문서와 진행 순서를 안내한다.
- 적용 범위: INGRADIENT 전체 프로젝트
- 상태: Draft
- Owner: Dev / QA
- 마지막 수정일: 2026-03-28

> 이 가이드를 따르기 전에 `common_guide.md`를 먼저 읽는다. 소통, 테스트, 코드 품질, 문서 원칙이 모든 작업에 공통 적용된다.

## 0단계: 테스트 요청 시 확인할 것

**중요: 마음대로 진행하지 않는다.** 애매하거나 판단이 필요한 부분이 있으면 반드시 질문한다. 더 나은 테스트 방법이 있다고 판단되면 먼저 제안하고 동의를 구한 뒤 진행한다.

**테스트는 생략하지 않는다.** 작성 가능한 모든 테스트를 반드시 작성한다. unit, integration, e2e 구분 없이 해당되면 다 한다.

테스트 요청을 받았을 때 아래 항목이 명시되지 않았으면 먼저 확인한다.

| 확인 항목 | 누락 시 질문 |
|-----------|-------------|
| 대상 프로젝트 | 어떤 프로젝트(서비스)를 테스트하는가? |
| 대상 기능 | 어떤 기능의 테스트인가? `features/{기능명}.md` 기획 문서가 있는가? |
| 구현 상태 | 구현이 완료된 상태인가, 구현 중인가? |
| 기존 테스트 | 기존 테스트가 있는가? 추가인가, 전면 작성인가? |

## 1단계: 테스트 전략 확인

- `../ops/test_strategy.md` — 테스트 원칙, 레벨별 기준, 도메인별 강조점
- `../dev/coding_convention.md` 7장 — 테스트 규칙

### 핵심 원칙 요약

- 작성 가능한 모든 테스트를 반드시 작성한다
- 백엔드, 프론트엔드, UI 구분 없이 다 한다
- bug fix에는 회귀 테스트를 반드시 남긴다

## 2단계: 기획 문서의 테스트 계획 확인

- `projects/{프로젝트}/features/{기능명}.md` — 테스트 계획 섹션
- `../product/test_plan_template.md` — 테스트 계획 양식 참고

기획 문서의 테스트 계획 섹션에서 아래를 확인한다.

| 항목 | 확인 내용 |
|------|-----------|
| 핵심 시나리오 | happy path, failure path, permission path가 정의되어 있는가 |
| 계약 영향 | API, data, auth 변경에 대한 검증 항목이 있는가 |
| 회귀 항목 | 기존 핵심 흐름 중 다시 확인할 것이 있는가 |
| 완료 기준 | 테스트 통과 기준, known issue, release blocker 조건이 있는가 |

## 3단계: 자동화 테스트 vs 수동 테스트 구분

테스트 항목을 작성하기 전에 자동화할 것과 사람이 직접 확인할 것을 구분한다.

| 구분 | 대상 | 작성 위치 |
|------|------|-----------|
| 자동화 테스트 | 로직, API, 데이터, 권한, 상태 전이 등 코드로 검증 가능한 것 | 프로젝트 코드 내 테스트 파일 |
| 수동 테스트 | 시각적 확인, UX 흐름, 환경 의존, 크로스 브라우저 등 사람이 직접 봐야 하는 것 | `projects/{프로젝트}/manual_tests.md` |

### 수동 테스트 항목 추가

새 기능이나 변경이 있으면 `manual_tests.md`에 수동 테스트 항목을 누적 추가한다.
- 기능명, 버전, 관련 기획 문서 링크를 함께 기록한다
- 자동화가 가능해진 항목은 자동화 전환 후 `manual_tests.md`에서 제거한다
- 수동 테스트 항목이 없으면 추가하지 않아도 된다 (모두 자동화 가능한 경우)

## 4단계: 자동화 테스트 작성

### 테스트 종류별 가이드

#### Unit Test

- 개별 함수, 서비스 로직의 입력/출력 검증
- 외부 의존성은 mock 처리
- 기존 테스트 패턴을 따른다 (프로젝트 내 기존 테스트 파일 참고)

#### Integration Test

- API 엔드포인트의 요청/응답 검증
- DB와 함께 실제 동작 확인
- 인증/인가 흐름 포함 (토큰 검증, 권한 체크)
- `../dev/api_convention.md` — 응답 구조, 에러 코드 기준

#### E2E / Scenario Test

- 사용자 시나리오 기반 전체 흐름 검증
- 기획 문서의 "기본 흐름"과 "예외 흐름"을 기준으로 작성
- 프론트엔드 UI 테스트 포함

#### 회귀 테스트

- 버그 수정 시 반드시 작성
- 버그를 재현하는 테스트를 먼저 작성하고, 수정 후 통과를 확인한다
- `../ops/debug_notes.md`에 기록된 사례가 있으면 참고

### 테스트 작성 원칙

- 테스트는 독립적으로 실행 가능해야 한다
- 테스트 간 순서 의존성을 만들지 않는다
- 테스트가 구현에 과도하게 결합되지 않게 한다 (내부 구현이 아닌 동작을 검증)
- 기존 프로젝트의 테스트 구조, 네이밍, 헬퍼를 따른다

## 5단계: 테스트 실행 및 검증

### 실행

- 모든 테스트를 로컬에서 실행하고 통과를 확인한다
- CI 파이프라인에서도 통과하는지 확인한다 (`../dev/ci_cd.md` 참고)

### 검증 체크리스트

- [ ] 기획 문서의 핵심 시나리오가 모두 테스트로 커버되는가
- [ ] happy path가 테스트되는가
- [ ] failure path (에러, 타임아웃, 잘못된 입력)가 테스트되는가
- [ ] permission path (권한 없음, 다른 역할)가 테스트되는가
- [ ] API 계약 (요청/응답 구조, 상태 코드)이 테스트되는가
- [ ] DB 변경이 있으면 migration 전후가 검증되는가
- [ ] 기존 테스트가 깨지지 않는가
- [ ] 버그 수정이면 회귀 테스트가 있는가

## 6단계: 테스트 완료 후

### 문서 업데이트

| 문서 | 수정 내용 |
|------|-----------|
| `releases/v{버전}.md` | 해당 항목 테스트 완료 체크 |
| `features/{기능명}.md` | 테스트 계획 섹션에 결과 반영 (known issue 등) |
