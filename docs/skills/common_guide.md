# Common Guide

- 문서명: Common Guide
- 목적: 모든 skills 가이드에서 공통으로 적용되는 원칙을 정의한다. 평소 대화에서도 항상 참고한다.
- 적용 범위: INGRADIENT 전체
- 상태: Draft
- Owner: Product / Dev
- 마지막 수정일: 2026-03-28

## 소통 원칙

- **마음대로 진행하지 않는다.** 애매하거나 판단이 필요한 부분이 있으면 반드시 질문한다.
- **가정을 숨기지 않는다.** 불확실한 것이 있으면 자신의 가정을 명시적으로 밝히고 확인을 받는다.
- **해석이 여러 개면 골라서 진행하지 않는다.** 선택지를 제시하고 어느 쪽인지 확인한다.
- **더 나은 방법이나 더 단순한 방법이 있으면 제안한다.** 동의를 구한 뒤 진행한다.
- **요청이 기존 문서나 컨벤션과 충돌하면 알린다.** 그냥 따르지 말고 충돌 사실을 공유하고 어떻게 할지 확인한다.
- **요청자의 말이 항상 맞다고 가정하지 않는다.** 문제가 보이면 정중하게 짚고 넘어간다.

## 테스트 원칙

- **테스트는 생략하지 않는다.** 작성 가능한 모든 테스트를 반드시 작성한다.
- 백엔드, 프론트엔드, UI 구분 없이 해당되면 다 한다.
- bug fix에는 회귀 테스트를 반드시 남긴다.
- 별도 요청이 없어도 항상 작성한다.

## 코드 품질 원칙

### 단순함 우선

- **요청받은 것만 구현한다.** 요청하지 않은 기능, 추상화, "향후 확장성"을 미리 넣지 않는다.
- **최소한의 코드로 문제를 해결한다.** 200줄로 쓴 걸 50줄로 할 수 있으면 다시 쓴다.
- **불가능한 시나리오에 대한 에러 처리를 넣지 않는다.** 실제로 발생하는 경우만 처리한다.
- "시니어 엔지니어가 보기에 과하다"고 느껴지면 단순화한다.

### 기본 규칙

- **파일 하나는 200줄 미만.** 넘으면 책임을 분리하여 파일을 쪼갠다.
- **하드코딩 금지.** 설정값, URL, 포트, 경로, 매직 넘버는 상수나 환경변수로 관리한다.
- **반복 금지.** 같은 로직이 2곳 이상이면 공통 함수/모듈로 추출한다.
- **장기적 관점.** 당장 동작하더라도 나중에 관리하기 어려운 구조는 피한다.
- 기존 코드 패턴과 컨벤션을 따른다. 불필요한 새 패턴을 도입하지 않는다.
- **UI는 `@ingradient/ui`를 사용한다.** 공용 디자인 시스템의 컴포넌트, 토큰, 패턴을 우선 사용한다. 커스텀 UI가 필요하면 먼저 `@ingradient/ui`에 없는지 확인한다.

## 문서 원칙

- **기능 하나 = 파일 하나.** 기획 문서는 `features/{기능명}.md` 한 파일에 모든 내용을 담는다.
- 여러 기능을 한번에 요청받더라도 기능별로 파일을 쪼개서 각각 작성한다.
- 기존 문서와 내용이 충돌하거나 중복되지 않도록 확인한다.
- 용어는 `terminology.md`와 일관되게 사용한다.

## 작업 흐름

- **각 단계가 끝나면 다음 단계로 넘어갈지 확인한다.**
  - 기획 → 구현할지 확인
  - 구현 → 테스트할지 확인
  - 테스트 → 코드 리뷰/배포로 진행
- **디버깅은 언제든 진입할 수 있다.** 구현 중, 테스트 중, 운영 중, 사용 중 어디서든 버그/장애가 발생하면 `debug_reading_guide.md`로 진입한다. 디버깅 완료 후에는 테스트 단계로 합류한다.
- **구현 중 기획에 빠진 부분을 발견하면** 기획 문서에 오픈 이슈로 기록하고, 필요하면 기획 단계로 돌아간다.
- **보고 전에 최종 검토한다.** 충돌, 중복, 용어 일관성, 링크 정확성을 확인한다.
- 문제가 발견되면 수정하고, 판단이 어려우면 보고 시 함께 언급한다.

## 목표 기반 실행

- **작업을 시작하기 전에 완료 기준을 먼저 정의한다.** "뭘 하면 끝인가"를 명확히 한다.
- 모호한 요청은 검증 가능한 목표로 변환한다.
  - "검증 추가해줘" → "잘못된 입력에 대한 테스트를 작성하고 통과시킨다"
  - "버그 수정해줘" → "버그를 재현하는 테스트를 작성하고 통과시킨다"
  - "리팩토링해줘" → "기존 테스트가 수정 전후 모두 통과하는 것을 확인한다"
- 여러 단계가 필요한 작업은 단계별 검증 기준을 세운다.
  - `1. [단계] → 확인: [검증 방법]`
  - `2. [단계] → 확인: [검증 방법]`
