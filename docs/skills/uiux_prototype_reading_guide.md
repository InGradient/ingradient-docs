- 문서명: UIUX Prototype Reading Guide
- 목적: UI/UX 프로토타입을 만들 때 읽어야 하는 문서와 진행 순서를 안내한다. 프로젝트 시작, 새 기능 기획, 아이디어 검증 등 UI 확인이 필요한 모든 상황에서 사용한다.
- 적용 범위: INGRADIENT 전체 프로젝트
- 상태: Draft
- Owner: Product / Dev
- 마지막 수정일: 2026-03-28

## 프로토타입이란

실질적으로 동작하지는 않지만 화면 구조와 UX 흐름을 확인할 수 있는 수준의 UI다. 실제 API 연동 없이 mock data로 화면을 채우고, 네비게이션과 주요 상태를 확인할 수 있다.

## 언제 사용하는가

| 상황 | 설명 |
|------|------|
| 프로젝트 시작 | 전체 화면 구조와 네비게이션을 먼저 검증할 때 |
| 새 기능 기획 | 기획 문서의 UI/UX 방향을 실물로 확인할 때 |
| 아이디어 검증 | 구현 전에 화면으로 아이디어를 빠르게 확인할 때 |
| UI 리뉴얼 | 기존 화면의 구조를 바꾸기 전에 새 구조를 먼저 확인할 때 |

## 0단계: 프로토타입 요청 시 확인할 것

**중요: 마음대로 진행하지 않는다.** UI 방향이 불확실하면 먼저 확인한다.

| 확인 항목 | 누락 시 질문 |
|-----------|-------------|
| 대상 프로젝트 | 어떤 프로젝트의 프로토타입인가? |
| 대상 화면 | 어떤 화면/기능의 프로토타입인가? 전체인가 부분인가? |
| 기획 문서 | `features/{기능명}.md`에 UI/UX 섹션이 있는가? |
| 기존 화면 | 기존에 만든 프로토타입이나 실제 화면이 있는가? 수정인가 신규인가? |
| 대상 사용자 | 이 화면을 사용하는 사용자 유형은? (관리자, 운영자, 일반 사용자 등) |

## 1단계: 기획 문서 확인

프로토타입의 기준이 되는 문서를 읽는다.

### 기능 기획 문서

- `projects/{프로젝트}/features/{기능명}.md` — UI/UX 방향 섹션
- 없으면 `feature_planning_reading_guide.md`를 따라 기획부터 작성한다

### UI/UX 참고 문서

- `../product/uiux_planning_template.md` — UI/UX 기획 양식 참고
- `../product/user_flow.md` — 전체 사용자 흐름에서 이 화면의 위치
- `../product/role_permission_flow.md` — 역할별 화면 접근 범위

### 디자인 시스템

- `../projects/ingradient-ui/features/design_tokens.md` — 디자인 토큰
- `../projects/ingradient-ui/features/generic_components.md` — 공용 컴포넌트
- `../projects/ingradient-ui/features/reusable_patterns.md` — 재사용 패턴

## 2단계: 프로토타입 작성

### 기본 규칙

- **`@ingradient/ui`를 사용한다.** 공용 디자인 시스템의 컴포넌트, 토큰, 패턴을 그대로 사용하여 실제 제품과 동일한 UI를 만든다. 커스텀 UI가 필요하면 먼저 `@ingradient/ui`에 없는지 확인한다
- **해당 프로젝트 repo 안에 만든다.** 별도 repo를 만들지 않는다
- **mock data를 사용한다.** 실제 API 연동 없이 정적 데이터로 화면을 채운다
- **네비게이션이 동작해야 한다.** 화면 간 이동은 실제로 되어야 한다 (라우팅)

### 포함할 것

| 항목 | 설명 |
|------|------|
| 화면 레이아웃 | 헤더, 사이드바, 메인 콘텐츠, 패널 등 정보 구조 |
| 네비게이션 | 화면 간 이동, 탭, 브레드크럼 |
| 데이터 표시 | mock data로 리스트, 상세, 테이블, 차트 등 |
| 폼/입력 | 입력 필드, 셀렉트, 버튼 (실제 제출은 안 됨) |
| 주요 상태 | empty, loading, success, error, permission denied |
| 반응형 | desktop 기준, 필요하면 tablet/compact |

### 포함하지 않을 것

| 항목 | 이유 |
|------|------|
| 실제 API 호출 | 프로토타입 목적에 불필요 |
| 실제 인증/권한 검증 | mock으로 대체 |
| 실제 DB 연동 | mock data 사용 |
| 비즈니스 로직 | 구현 단계에서 진행 |

### mock data 작성 원칙

- 실제와 유사한 데이터를 사용한다 (lorem ipsum이 아닌 도메인 맥락에 맞는 데이터)
- 다양한 케이스를 포함한다 (데이터 많을 때, 적을 때, 없을 때, 긴 텍스트, 짧은 텍스트)
- mock data는 별도 파일로 분리한다 (나중에 실제 API로 교체하기 쉽도록)

## 3단계: 상태 설계 확인

각 화면에서 아래 상태가 모두 확인 가능한지 점검한다.

| 상태 | 확인 내용 |
|------|-----------|
| empty | 데이터가 없을 때 안내 메시지가 보이는가 |
| loading | 로딩 중 표시가 있는가 |
| success | 정상 데이터가 올바르게 표시되는가 |
| error | 에러 발생 시 사용자가 이해 가능한 메시지가 보이는가 |
| partial failure | 일부만 실패했을 때 어떻게 보이는가 |
| permission denied | 권한이 없을 때 기능이 숨겨지거나 안내가 보이는가 |

## 4단계: 프로토타입 검증

### 기획 문서와 대조

- features/ 기획 문서의 UI/UX 섹션에 명시된 화면이 모두 만들어졌는가
- 기획의 핵심 행동 (primary action, secondary action)이 화면에 반영되었는가
- 기획의 사용자 시나리오대로 네비게이션이 흐르는가

### 차이 발견 시

- 프로토타입이 기획보다 나으면 → 기획 문서를 업데이트한다
- 기획이 프로토타입보다 나으면 → 프로토타입을 수정한다
- 판단이 어려우면 → 두 가지를 함께 보여주고 확인한다

### 발견한 문제 기록

- UI 문제를 `features/{기능명}.md` 오픈 이슈에 기록한다
- `@ingradient/ui`에 없는 컴포넌트가 필요하면 기록한다

## 체크리스트

- [ ] `@ingradient/ui` 컴포넌트를 사용했는가
- [ ] mock data로 화면이 채워져 있는가
- [ ] mock data가 실제 도메인 맥락에 맞는가
- [ ] 네비게이션이 동작하는가
- [ ] 주요 상태 (empty, loading, success, error, permission denied)가 확인 가능한가
- [ ] 기획 문서의 UI/UX 섹션과 일치하는가
- [ ] 기획과 차이가 있으면 문서가 업데이트되었는가
- [ ] 발견한 문제가 오픈 이슈에 기록되었는가
