# Idea

$ARGUMENTS 아이디어를 체계적으로 기록하고 정리한다.

**이 스킬은 아래 상황에서 사용한다:**
- 새로운 기능 아이디어가 떠올랐을 때
- 고객 요구사항이나 피드백이 들어왔을 때
- 기존 기능의 개선 방향이 생겼을 때
- 기술 부채나 구조 개선이 필요할 때
- 아직 어떤 버전에 넣을지 정해지지 않은 것들

## 0단계: 사전 확인

| 확인 항목 | 누락 시 질문 |
|-----------|-------------|
| 대상 프로젝트 | 어떤 프로젝트에 해당하는 아이디어인가? 여러 프로젝트에 걸치는가? |
| 아이디어 유형 | 새 기능 / 기존 기능 개선 / 기술 부채 / 고객 요구 / UX 개선 중 어느 것인가? |
| 긴급도 | 지금 바로 다음 버전에 넣어야 하는가, 나중에 검토할 것인가? |
| 배경 | 이 아이디어가 나온 이유는? (고객 피드백, 운영 이슈, 경쟁사 분석 등) |

## 1단계: 기존 문서 확인

아이디어가 이미 기록되어 있거나 진행 중인지 먼저 확인한다.

| 확인 대상 | 읽을 문서 |
|-----------|-----------|
| 이미 기획된 기능인지 | projects/{프로젝트}/features/README.md |
| 이미 로드맵에 있는지 | projects/{프로젝트}/roadmap.md |
| 이미 upcoming에 있는지 | projects/{프로젝트}/upcoming.md |
| 현재 릴리즈 계획에 있는지 | projects/{프로젝트}/releases/README.md |

이미 있으면 기존 항목에 추가 맥락이나 우선순위 변경을 반영한다. 새로 만들지 않는다.

## 2단계: 적합성 검증

기획까지는 하지 않지만, 기록 전에 현재 상태에서 문제가 없는지 확인한다.

**방향 확인:**

| 확인 관점 | 읽을 문서 |
|-----------|-----------|
| 전체 시스템 목적과 범위 | docs/architecture/system_overview.md |
| 서비스별 책임 경계 | docs/architecture/service_responsibility.md |
| 대상 프로젝트의 목적과 비범위 | projects/{프로젝트}/project_overview.md |

**충돌 확인:**

| 확인 관점 | 읽을 문서 |
|-----------|-----------|
| 기존 기능 목록 | projects/{프로젝트}/features/README.md |
| 사용자 흐름에서의 위치 | docs/product/user_flow.md |
| 권한 체계에 영향 | docs/product/role_permission_flow.md |
| 용어 충돌 | docs/product/terminology.md |
| 다른 서비스에 영향 | docs/product/multi_product_integration.md |

**검증 결과에 따라:**
- 방향에 맞고 충돌 없음 → 3단계로 진행
- 방향에 맞지만 기존 기능과 겹침 → 기존 기능 개선으로 전환
- 충돌 있음 → 충돌 사실을 알리고 확인
- 방향에 맞지 않음 → 솔직히 말하고 대안 제안

검증이 끝나면 로드맵 중 어디에 들어가면 좋을지 제안한다.

## 3단계: upcoming.md에 기록

projects/{프로젝트}/upcoming.md에 아이디어를 기록한다.

**기록 형식 (유형별):**

    ## 새 기능 아이디어
    | 우선순위 | 아이디어 | 배경 | 대상 버전 |
    |----------|----------|------|-----------|
    | 높음 | 기능 이름 | 왜 필요한지 한 줄 | 미정 |

    ## 개선사항
    | 우선순위 | 항목 | 배경 | 대상 버전 |
    |----------|------|------|-----------|
    | 중간 | 개선 항목 | 이유 | 미정 |

    ## 기술 부채
    | 우선순위 | 항목 | 영향 | 대상 버전 |
    |----------|------|------|-----------|
    | 낮음 | 부채 항목 | 방치 시 영향 | 미정 |

    ## 고객 요구사항
    | 우선순위 | 항목 | 요청 출처 | 대상 버전 |
    |----------|------|-----------|-----------|
    | 높음 | 요구사항 | 고객/채널 | 미정 |

**기록 원칙:**
- **한 줄로 핵심을 적는다.** 상세 기획은 나중에 한다. 지금은 "뭘, 왜"만 적는다
- **배경을 반드시 남긴다.** 나중에 왜 이 아이디어가 나왔는지 모르면 판단할 수 없다
- **우선순위를 매긴다.** 높음/중간/낮음. 판단이 어려우면 질문한다
- **대상 버전은 "미정"으로 둔다.** 릴리즈 계획에 편입될 때 버전이 정해진다

## 4단계: roadmap 반영

아이디어가 방향성에 영향을 주면 roadmap도 업데이트한다.

| 조건 | roadmap 반영 |
|------|-------------|
| 새로운 단계나 방향이 생기면 | 단계별 계획에 추가 |
| 기존 우선순위가 바뀌면 | 우선순위 섹션 업데이트 |
| 새 기술 부채가 발견되면 | 기술 부채 섹션에 추가 |
| 사소한 개선이면 | roadmap에 넣지 않음 (upcoming에만) |

## 5단계: 분석 및 제안

아이디어를 기록한 후 적극적으로 제안한다.

- 비슷한 아이디어가 이미 있으면 통합을 제안한다
- 아이디어가 기존 아키텍처나 서비스 책임과 충돌하면 알린다
- 구현 난이도 대비 효과가 낮으면 솔직히 말한다
- 더 단순한 대안이 있으면 제시한다

## 규칙

- **마음대로 기획까지 진행하지 않는다.** 이 스킬은 기록과 정리까지만 한다
- **이미 있는 아이디어를 중복 생성하지 않는다.** 기존 항목에 맥락을 추가한다
- **배경 없이 기록하지 않는다.** "왜"가 없으면 나중에 판단할 수 없다
- **방향에 맞지 않으면 솔직히 말한다.** 무조건 수용하지 않는다

## 완료 기준

- [ ] 기존 문서에 이미 있는 아이디어가 아닌지 확인했다
- [ ] 적합성 검증을 완료했다 (방향, 충돌)
- [ ] upcoming.md에 기록했다 (뭘, 왜, 우선순위)
- [ ] roadmap 업데이트가 필요하면 반영했다
- [ ] 로드맵 위치를 제안했다
