## 0단계: 커밋 전 확인할 것

**중요: 마음대로 커밋하지 않는다.** 커밋은 되돌리기 어려운 작업이므로 충분히 확인한 뒤 진행한다.

| 확인 항목 | 누락 시 질문 |
|-----------|-------------|
| 대상 브랜치 | 어떤 브랜치에서 작업하고 있는가? main에 직접 커밋하는 건 아닌가? |
| 작업 완료 여부 | 구현과 테스트가 모두 완료되었는가? |
| 커밋 범위 | 한 번에 커밋할 범위가 적절한가? 여러 기능을 섞고 있진 않은가? |

## 1단계: 컨벤션 위반 검사

커밋 전에 코드가 컨벤션을 지키고 있는지 점검한다. 위반이 발견되면 **지금 수정한다.** 나중에 하지 않는다.

### 코딩 컨벤션 (`../dev/coding_convention.md`)

- [ ] 파일 네이밍이 kebab-case인가
- [ ] 타입/클래스는 PascalCase, 함수/변수는 camelCase인가
- [ ] 파일당 200줄 미만인가. 넘으면 분리했는가
- [ ] 레이어 의존 방향이 단방향인가
- [ ] 경계 입력에 runtime validation이 있는가
- [ ] 하드코딩이 없는가 (설정값, URL, 매직 넘버)
- [ ] 에러 처리가 적절한가 (silent catch, floating promise 없음)
- [ ] 로그에 모듈명과 관련 ID가 포함되어 있는가

### API 컨벤션 (`../dev/api_convention.md`) — API 변경이 있으면

- [ ] 리소스 네이밍이 규칙을 따르는가
- [ ] 요청/응답 구조가 일관되는가
- [ ] 에러 응답 형식이 맞는가
- [ ] 페이지네이션 패턴이 맞는가

### 정밀한 변경 원칙 (`implementation_reading_guide.md`)

- [ ] 요청과 관련 없는 코드를 건드리지 않았는가
- [ ] 내 변경으로 생긴 고아 코드만 정리했는가
- [ ] 변경한 모든 줄이 요청과 직접 연결되는가

## 2단계: 테스트 통과 확인

커밋 전에 모든 테스트가 통과하는지 확인한다.

### 로컬 실행

```
lint → typecheck → unit test → integration test
```

- [ ] lint가 통과하는가
- [ ] typecheck가 통과하는가
- [ ] 기존 테스트가 깨지지 않는가
- [ ] 새로 작성한 테스트가 통과하는가

### 테스트 커버리지

- [ ] 기획 문서의 핵심 시나리오가 테스트로 커버되는가
- [ ] happy path, failure path, permission path가 있는가
- [ ] 버그 수정이면 회귀 테스트가 있는가

위반이나 실패가 있으면 **커밋하지 않고 먼저 수정한다.**

## 3단계: 문서 업데이트 확인

코드와 문서가 일치하는지 확인한다.

- [ ] `features/{기능명}.md` — 구현 중 변경된 사항이 반영되었는가
- [ ] `releases/v{버전}.md` — 완료한 항목이 체크되었는가 (`[x]`)
- [ ] `features/README.md` — 새 기능이 등록되었는가
- [ ] `api_contract.md` — API 변경이 반영되었는가
- [ ] `data_model.md` — DB 변경이 반영되었는가
- [ ] `terminology.md` — 새 용어가 추가되었는가

## 4단계: 커밋

### 브랜치 확인

- `../dev/git_workflow.md`의 브랜치 네이밍을 따르는가
- main이 아닌 작업 브랜치에서 작업하고 있는가

| 접두사 | 용도 |
|--------|------|
| feat/ | 새 기능 |
| fix/ | 버그 수정 |
| refactor/ | 리팩토링 |
| docs/ | 문서 변경 |
| test/ | 테스트 추가/수정 |
| chore/ | 설정, 의존성, CI |

### 커밋 메시지

```
{type}: {한 줄 설명}

{본문 (선택 — 왜 이 변경을 했는지)}
```

- 의미 단위로 커밋한다. 한 커밋에 여러 목적을 섞지 않는다
- "무엇을 했는가"보다 "왜 했는가"를 우선한다
- breaking change는 본문에 `BREAKING CHANGE:` 접두사로 명시한다

### 커밋 단위 가이드

| 상황 | 커밋 단위 |
|------|-----------|
| DB migration | migration 별도 커밋 |
| 백엔드 로직 + API | 함께 커밋 가능 |
| 프론트엔드 UI | 화면 단위 또는 기능 단위 |
| 테스트 | 구현과 함께 또는 별도 커밋 |
| 문서 업데이트 | 코드와 함께 또는 별도 커밋 |
| 컨벤션 위반 수정 | 기능 커밋과 별도 커밋 (섞지 않는다) |

## 5단계: PR 생성

### PR 작성

- 제목은 커밋 메시지 형식과 동일하게 작성한다
- 본문에 아래를 포함한다:
  - **변경 이유** — 왜 이 변경이 필요한가
  - **주요 변경사항** — 무엇이 바뀌었는가
  - **테스트 방법** — 어떻게 검증했는가
  - **관련 문서** — `features/{기능명}.md` 링크
- breaking change가 있으면 영향 범위를 명시한다
- migration이 있으면 실행 순서를 명시한다

### PR 체크리스트

PR 본문에 아래 체크리스트를 포함한다.

```markdown
## Checklist
- [ ] 컨벤션 위반 없음 (coding_convention, api_convention)
- [ ] 모든 테스트 통과
- [ ] 기획 문서와 일치
- [ ] releases 체크리스트 업데이트
- [ ] API/DB 변경 시 관련 문서 업데이트
- [ ] breaking change 없음 (있으면 명시)
- [ ] migration 없음 (있으면 순서 명시)
```

## 6단계: CI 확인 및 머지

### CI 통과 확인

- PR 생성 후 CI 파이프라인이 자동 실행된다 (`../dev/ci_cd.md` 참고)
- lint → test → build 순서로 통과해야 한다
- CI 실패 시 머지하지 않고 원인을 수정한다

### 리뷰 대응

- 리뷰 피드백이 있으면 수정 후 다시 커밋한다
- Block 수준 피드백은 반드시 해결 후 머지한다
- 리뷰 기준은 `../dev/code_review_guide.md` 참조

### 머지

- 리뷰 승인 + CI 통과 후 머지한다
- squash merge를 기본으로 한다
- 머지 후 작업 브랜치를 삭제한다

## 7단계: 머지 후 확인

### 배포 상태 확인

- main에 머지되면 자동 배포가 진행되는지 확인한다
- 배포 실패 시 즉시 revert 또는 hotfix를 진행한다

### 문서 최종 확인

- [ ] `releases/v{버전}.md`의 체크리스트가 최신 상태인가
- [ ] `release_notes.md`에 반영이 필요하면 업데이트했는가

## 다음 단계

머지가 완료되면 다음 작업으로 진행할지 확인한다.

- **같은 버전에 다음 기능이 있으면** → `releases/v{버전}.md`에서 다음 항목 확인 후 `implementation_reading_guide.md`를 따른다
- **버전이 완료되었으면** → `release_notes.md` 작성, 배포 확인
