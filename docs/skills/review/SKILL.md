# Review

$ARGUMENTS 범위의 코드를 기획 문서와 1:1 대조하여 빠뜨린 것이 없는지 검증한다.

- `total` → 프로젝트의 모든 features/ 문서와 코드를 전체 대조
- `{feature명}` → features/{feature명}.md와 해당 코드만 대조

## 0단계: 범위 결정

**$ARGUMENTS가 `total`이면:**
- features/ 폴더의 모든 기능 문서를 수집한다
- **한번에 전부 하지 않는다.** feature 하나씩 순서대로 리뷰한다
- 각 feature 리뷰가 끝나면 결과를 요약하고 "다음 feature를 리뷰할까요?" 라고 질문한다
- 모든 feature 리뷰가 끝나면 전체 결과를 종합 요약한다

**$ARGUMENTS가 특정 feature명이면:**
- features/$ARGUMENTS.md만 읽고 해당 코드만 검증한다

## 1단계: 문서 읽기

기능별로 아래 문서를 읽는다. **문서가 존재하지 않으면 건너뛰지 말고 질문한다.**

| 문서 | 목적 |
|------|------|
| features/{기능명}.md | 비즈니스 규칙, 상태 전이, 에러 케이스, 권한, UI/UX |
| wireframes/ 관련 화면 | 화면 구조, 필드, 인터랙션, 상태별 화면 |
| api_contract.md | API endpoint, 요청/응답, 에러 코드 |
| data_model.md | DB 스키마, 관계, 제약조건 |
| docs/product/role_permission_flow.md | 역할별 접근 권한 |

## 2단계: 검증 항목 추출

기획 문서에서 **검증 가능한 모든 항목**을 꺼내서 체크리스트로 나열한다. "문서와 대조한다"로 끝내지 않는다.

**추출 대상:**

비즈니스 규칙:
- [ ] 문서에 명시된 모든 비즈니스 규칙이 코드에 구현되어 있는가
- [ ] 각 규칙의 조건과 결과가 코드와 일치하는가
- [ ] 예외 케이스가 처리되어 있는가

상태 전이:
- [ ] 문서에 명시된 모든 상태 전이가 코드에 구현되어 있는가
- [ ] 허용되지 않는 상태 전이가 차단되어 있는가
- [ ] 상태 전이 시 연쇄 동작 (세션 종료, 알림 발송 등)이 구현되어 있는가

에러 케이스:
- [ ] 문서에 명시된 모든 에러 케이스가 코드에서 처리되는가
- [ ] 에러 코드와 메시지가 문서와 일치하는가
- [ ] 에러 발생 시 UI 동작 (토스트, 필드 에러, 다이얼로그)이 문서와 일치하는가

권한:
- [ ] 각 기능/API/화면의 접근 권한이 role_permission_flow.md와 일치하는가
- [ ] 권한 없는 사용자에게 기능이 숨겨지거나 비활성화되는가
- [ ] 권한 검증이 프론트엔드와 백엔드 양쪽에서 이루어지는가

API:
- [ ] api_contract.md의 모든 endpoint가 구현되어 있는가
- [ ] 요청/응답 필드가 api_contract.md와 일치하는가
- [ ] 에러 응답이 api_contract.md와 일치하는가
- [ ] pagination, filter, sort가 문서대로 동작하는가

DB:
- [ ] data_model.md에 명시된 테이블/컬럼이 실제 스키마와 일치하는가
- [ ] 관계 (FK, join)가 올바르게 설정되어 있는가
- [ ] 제약조건 (unique, not null, enum)이 문서와 일치하는가

UI:
- [ ] wireframe의 모든 필드/컬럼/버튼/뱃지가 렌더링되는가
- [ ] 인터랙션이 wireframe대로 동작하는가 (함수 내부까지 추적)
- [ ] 상태별 화면 (empty, loading, error, permission denied)이 구현되어 있는가
- [ ] 값 표시 형식 (날짜, 상태, 역할)이 문서와 일치하는가

테스트:
- [ ] 기획 문서의 핵심 시나리오가 테스트로 커버되는가
- [ ] 에러 케이스가 테스트에 포함되어 있는가
- [ ] 권한별 접근 테스트가 있는가

## 3단계: 코드 검증

체크리스트의 각 항목을 **코드를 직접 읽어서** 검증한다.

- **PASS** — 문서와 일치한다
- **FAIL** — 문서와 다르다 (빠뜨렸거나 잘못 구현)
- **WARN** — 문서에 명시되지 않았지만 확인이 필요하다

**검증 시 주의:**
- 함수가 "존재"하는 것만으로 PASS하지 않는다. 내부 로직까지 확인한다
- 하드코딩된 값이 사용자 입력을 대체하고 있지 않은지 확인한다
- prop이 선언만 되고 전달되지 않는 경우가 없는지 확인한다
- 에러 처리가 silent catch로 무시되고 있지 않은지 확인한다

## 5단계: FAIL 항목 수정

사용자가 동의하면 FAIL 항목을 수정한다.

- 수정 후 해당 항목을 다시 검증한다
- 수정 범위를 최소화한다 — 리뷰에서 발견한 누락만 수정, 리팩토링을 섞지 않는다
- 기획 문서 자체를 수정해야 하면 **반드시 먼저 확인을 받는다**

## 규칙

- **문서의 모든 항목을 빠짐없이 검증한다.** "대체로 맞아 보인다"로 넘어가지 않는다
- **코드를 직접 읽는다.** 파일명이나 함수명만 보고 판단하지 않는다
- **함수 내부까지 추적한다.** 함수가 존재하는 것과 올바르게 동작하는 것은 다르다
- **기획 문서가 기준이다.** 코드가 다르면 코드를 고친다
- **추측으로 PASS하지 않는다.** 확인이 안 되면 WARN으로 기록한다

## 완료 기준

- [ ] 범위 내 모든 features/ 문서의 항목이 검증되었다
- [ ] 모든 FAIL 항목이 수정되었거나 수정 계획이 합의되었다
- [ ] 모든 WARN 항목에 대해 확인을 요청했다
- [ ] 수정된 코드가 테스트를 통과한다
