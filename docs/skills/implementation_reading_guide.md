- 목적: 기능 구현을 요청받았을 때 읽어야 하는 문서와 진행 순서를 안내한다.
- 적용 범위: INGRADIENT 전체 프로젝트
- 상태: Draft
- Owner: Dev
- 마지막 수정일: 2026-03-28

> 이 가이드를 따르기 전에 `common_guide.md`를 먼저 읽는다. 소통, 테스트, 코드 품질, 문서 원칙이 모든 작업에 공통 적용된다.

## 0단계: 구현 요청 시 확인할 것

구현 요청을 받았을 때 아래 항목이 명시되지 않았으면 먼저 확인한다.

**중요: 마음대로 진행하지 않는다.** 애매하거나 판단이 필요한 부분이 있으면 반드시 질문한다. 더 나은 구현 방법이 있다고 판단되면 먼저 제안하고 동의를 구한 뒤 진행한다. 요청 내용이 기존 코드 패턴이나 컨벤션과 충돌하는 경우에도 그냥 따르지 말고 충돌 사실을 알리고 어떻게 할지 확인한다.

| 확인 항목 | 누락 시 질문 |
|-----------|-------------|
| 대상 프로젝트 | 어떤 프로젝트(서비스)에서 구현하는가? |
| 기획 문서 | `features/{기능명}.md` 기획 문서가 있는가? 없으면 먼저 기획부터 해야 하는가? |
| 대상 버전 | 어떤 릴리즈 버전에 포함되는 구현인가? |
| 구현 범위 | 기획 문서 전체를 구현하는가, 일부만 구현하는가? |
| 기존 코드 | 수정할 기존 코드가 있는가, 새로 작성하는가? |

**테스트는 확인 대상이 아니다.** 구현 시 작성 가능한 모든 테스트(unit, integration, e2e)를 반드시 작성한다. 별도 요청이 없어도 생략하지 않는다.

## 1단계: 기획 문서 확인

구현 대상의 기획 문서를 먼저 읽는다.

- `projects/{프로젝트}/features/{기능명}.md` — 기능 명세, 시나리오, UI/UX, 테스트 계획
- `projects/{프로젝트}/releases/v{버전}.md` — 이번 버전에서의 위치, 의존성, 배포 계획

기획 문서가 없으면 `feature_planning_reading_guide.md`를 따라 기획부터 진행한다.

## 2단계: 코딩 컨벤션

구현 전에 반드시 읽는 문서들이다.

### 필수

- `../dev/coding_convention.md` — 파일 구조, 네이밍 (kebab-case 파일, PascalCase 타입, camelCase 함수), 레이어 의존성, 에러 처리, 로깅, 테스트 규칙
- `../dev/git_workflow.md` — 브랜치 생성, 커밋, PR, 머지 절차

### API 작업이 있으면

- `../dev/api_convention.md` — 리소스 네이밍, 요청/응답 구조, 페이지네이션, 스키마 검증, 비동기 job 패턴, 에러 상태 코드

### DB/스키마 변경이 있으면

- `../dev/migration_policy.md` — backward compatibility, 배포 순서 (expand → dual-write → rollout → cleanup), rollback 전략

## 3단계: 대상 프로젝트 코드 파악

구현할 프로젝트의 현재 코드 구조를 파악한다.

### 프로젝트 문서

- `projects/{프로젝트}/architecture.md` — 내부 구조, 모듈 구성, 기술 스택
- `projects/{프로젝트}/data_model.md` — DB 엔티티, 관계, 상태값

### 코드베이스

- 프로젝트 루트의 디렉토리 구조 확인
- 구현할 기능과 관련된 기존 코드 탐색 (유사 기능, 공통 유틸, 패턴)
- 기존 코드의 네이밍, 구조, 패턴을 따른다. 새로운 패턴을 도입하지 않는다

## 4단계: 연관 서비스 확인

다른 서비스에 영향을 주는 변경이면 확인한다.

| 조건 | 읽을 문서 |
|------|-----------|
| API 계약이 바뀌면 | `projects/{프로젝트}/api_contract.md`, `../api/api_principles.md` |
| 권한 체계가 바뀌면 | `../product/role_permission_flow.md` |
| 다른 서비스와 연동이 있으면 | `../product/multi_product_integration.md`, 연동 대상의 `api_contract.md` |
| auth 토큰/멤버십이 관련되면 | `../api/auth_api.md` |
| Edge 동기화가 관련되면 | `../api/edge_api.md` |

## 5단계: 구현

### 문서와 코드의 관점 차이

- **문서(features/)는 기능별로 나뉜다.** 사용자/기획자 관점에서 이해하기 쉽도록 "로그인", "초대", "라이선스" 등 기능 단위로 작성한다.
- **코드(modules/)는 도메인 모듈별로 나뉜다.** 재사용과 유지보수를 위해 auth, users, memberships, licenses 등 책임 단위로 나눈다.
- **하나의 기능은 여러 모듈의 조합으로 구현한다.** 예: "사용자 생성" 기능은 users.service + memberships.service + audit.service를 조합한다. 하나의 거대한 함수로 만들지 않는다.
- **겹치는 로직은 공통 함수로 추출한다.** 여러 기능에서 같은 동작이 필요하면 모듈의 공통 서비스로 만들고 호출한다. 기능별로 따로 구현하지 않는다.

### 구현 원칙

- 기획 문서의 범위를 벗어나지 않는다
- 기존 코드 패턴과 컨벤션을 따른다
- 한 번에 너무 많은 것을 바꾸지 않는다
- 구현 중 기획에 빠진 부분을 발견하면 기획 문서에 오픈 이슈로 기록한다. 범위가 크면 구현을 멈추고 `feature_planning_reading_guide.md`로 돌아간다

### 정밀한 변경 원칙

기존 코드를 수정할 때는 요청받은 범위만 정확히 변경한다.

- **요청과 관련 없는 코드를 "개선"하지 않는다.** 주변 코드, 주석, 포맷팅을 건드리지 않는다.
- **망가지지 않은 것을 리팩토링하지 않는다.** 기존 스타일이 마음에 안 들어도 맞춘다.
- **관련 없는 dead code를 발견하면 삭제하지 말고 언급만 한다.**
- **내 변경으로 생긴 고아 코드(미사용 import, 변수, 함수)만 정리한다.**
- 변경한 모든 줄이 요청과 직접적으로 연결되는지 스스로 확인한다

### 코드 품질 원칙

운영과 코드 관리를 위해 아래를 반드시 지킨다.

**파일 크기와 구조:**
- 파일 하나는 200줄 미만을 유지한다. 넘어가면 책임을 분리하여 파일을 쪼갠다
- 폴더 구조를 바꾸거나 파일을 분리하는 것을 두려워하지 않는다. 구조 변경이 장기적으로 더 나으면 적극적으로 한다
- 파일 하나는 책임 하나를 가진다 (`coding_convention.md` 1장 참조)

**확장성과 유지보수:**
- 장기적인 효율성과 확장성을 항상 고려한다. 당장 동작하더라도 나중에 관리하기 어려운 구조는 피한다
- 하드코딩을 하지 않는다. 설정값, URL, 포트, 경로, 매직 넘버 등은 상수나 환경변수로 관리한다
- 같은 로직이 2곳 이상에서 반복되면 공통 함수/모듈로 추출한다
- 비즈니스 규칙과 인프라 코드를 분리한다 (DB 접근, API 호출 등은 별도 레이어)

**운영 편의:**
- 에러 메시지는 원인을 파악할 수 있도록 구체적으로 작성한다
- 로그에는 모듈명과 관련 ID를 포함한다 (`coding_convention.md` 6장 참조)
- 상태값, 에러 코드는 enum 또는 상수로 정의한다. 문자열을 직접 비교하지 않는다
- 외부 의존성(API, DB, 파일시스템)은 실패할 수 있으므로 적절한 에러 처리를 한다

### 구현 순서 (권장)

1. DB/스키마 변경 (migration)
2. 백엔드 로직 (service, route)
3. API 엔드포인트
4. 프론트엔드 UI
5. 테스트

## 6단계: 문서 업데이트

**구현이 끝난 후가 아니라, 구현하면서 계속 업데이트한다.** 기능 하나를 완료할 때마다 바로 반영한다. 마지막에 몰아서 하지 않는다.

### 구현 중 계속 업데이트

| 문서 | 시점 | 수정 내용 |
|------|------|-----------|
| `releases/v{버전}.md` | 기능/항목 완료할 때마다 | 체크리스트 완료 표시 (`[x]`), 구현 중 발견한 이슈나 변경사항 기록 |
| `features/{기능명}.md` | 구현 중 기획과 달라질 때마다 | 변경된 사항 반영, 오픈 이슈 추가/해결 |

### 변경 내용에 따라 업데이트

| 조건 | 수정할 문서 |
|------|-------------|
| API가 추가/변경됨 | `projects/{프로젝트}/api_contract.md` |
| DB 모델이 바뀜 | `projects/{프로젝트}/data_model.md` |
| 새 화면/기능이 추가됨 | `projects/{프로젝트}/product_guide.md` — 구현 중 기획과 달라진 부분 반영 |
| 화면 구조가 바뀜 | `projects/{프로젝트}/wireframes/{화면명}.md` — 실제 구현과 맞게 수정 |
| 사용자 흐름이 바뀜 | `projects/{프로젝트}/user_scenarios.md` — 실제 구현과 맞게 수정 |
| 새 용어가 생김 | `../product/terminology.md` |

### 릴리즈 전 업데이트

| 문서 | 수정 내용 | 시점 |
|------|-----------|------|
| `release_notes.md` | 변경 사항 요약 | 배포 후 |
| `user_guide.md` | 완성된 기능의 구체적 사용법 (스크린샷 등) | 개발 완료 후 |
| `operations.md` | 모니터링, 알람, 트러블슈팅 포인트 | 배포 전 |
| `manual_tests.md` | 수동 테스트 항목 누적 추가 | 테스트 전 |

**product_guide.md vs user_guide.md 구분:**
- `product_guide.md` — 개발 **전/중**에 작성. "뭘 할 수 있는가" (기획 입력). 구현 중 달라진 점 반영
- `user_guide.md` — 개발 **후**에 작성. "어떻게 쓰면 되는가" (실제 사용법)
