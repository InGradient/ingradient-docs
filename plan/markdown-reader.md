# Markdown 문서 리더 기획

## 1. 목적

이 문서는 `ingradient-docs` 저장소의 Markdown 문서를 별도 하드코딩 없이 탐색하고 읽을 수 있는 **문서 리더 UI**의 구현 방향을 정리한 기획 문서다.

이번 기획의 목표는 아래와 같다.

* 문서 메뉴를 수동으로 관리하지 않는다.
* 지정한 루트 폴더를 스캔해서 폴더/파일 구조를 자동으로 노출한다.
* 폴더를 누르면 하위 목록이 보이고, Markdown 파일을 누르면 본문이 열린다.
* UI는 `ingradient-ui` 기반으로 구현한다.
* 모바일에서도 탐색과 읽기가 자연스러워야 한다.
* `npm run dev` 실행 시 외부 기기에서 접속 가능해야 한다.

---

## 2. 핵심 결론

이번 문서 리더는 복잡한 수동 메뉴 시스템이 아니라, 아래 한 줄로 정리되는 구조가 맞다.

**지정된 콘텐츠 루트 폴더를 자동 스캔해서, 폴더는 목록으로 보여주고 `.md` 파일은 Markdown 본문으로 렌더링하는 파일시스템 기반 리더를 만든다.**

즉, 사용자가 말한 방향이 맞다.

* 루트 폴더 안에 새 폴더를 넣으면 자동으로 폴더 목록에 보인다.
* 루트 폴더 안에 새 `.md` 파일을 넣으면 자동으로 문서 목록에 보인다.
* 별도의 좌측 메뉴 JSON, 문서 인덱스 하드코딩, 수동 링크 추가를 기본 전제로 두지 않는다.

---

## 3. 왜 하드코딩을 피해야 하는가

`ingradient-docs` 는 이미 디렉터리 구조가 문서 분류 체계 역할을 하고 있다.

현재 구조 예시

* `docs/architecture`
* `docs/api`
* `docs/data`
* `docs/dev`
* `docs/governance`
* `docs/ops`
* `docs/product`
* `docs/projects/...`

이 구조에서는 사람이 메뉴를 하나씩 관리하는 방식보다, **폴더 구조 자체를 메뉴로 해석하는 방식**이 더 정확하고 유지보수 비용이 낮다.

하드코딩 메뉴 방식의 문제

* 새 문서를 추가할 때 메뉴도 같이 수정해야 한다.
* 실제 파일 구조와 메뉴 구조가 쉽게 어긋난다.
* 누락 문서가 생기기 쉽다.
* 모바일까지 고려하면 수동 메뉴 관리 비용이 더 커진다.

따라서 이 문서 리더는 “문서 CMS”가 아니라 “파일시스템 브라우저 + Markdown 리더”에 가깝게 설계하는 것이 맞다.

## 4. 콘텐츠 루트 정의

### 4.1 1차 대상 루트

1차 구현의 기본 콘텐츠 루트는 아래로 둔다.

* `/home/june/workspace/projects/ingradient-docs/docs`

즉, 문서 리더의 기본 대상은 `ingradient-docs/docs/` 전체다.

### 4.2 확장 가능성

향후 필요하면 아래도 같은 방식으로 추가할 수 있다.

* `plan/`
* 다른 문서 폴더

다만 1차는 복잡도를 줄이기 위해 **루트 1개(`docs/`)만 우선 지원**하는 것이 적절하다.

### 4.3 표시 규칙

루트 아래에서 다음 규칙으로 표시한다.

* 폴더는 목록에 표시
* `.md` 파일만 표시
* 그 외 파일 형식은 기본 비노출
* 숨김 파일과 숨김 폴더는 기본 비노출
* 정렬은 `폴더 먼저, 파일 나중, 이름순`

## 5. 화면 구성

이번 문서 리더는 화면을 많이 만들 필요가 없다.

필요한 화면은 아래 두 개면 충분하다.

* **폴더 목록 화면**
* **Markdown 읽기 화면**

두 화면 모두 상단에는 같은 breadcrumb 를 둔다.

### 5.1 공통 상단 구조

두 화면 공통으로 상단에 아래 요소를 둔다.

* breadcrumb
* 현재 경로 기준 제목
* 모바일용 트리 열기 버튼

즉, 화면이 바뀌어도 상단 탐색 방식은 동일해야 한다.

추가 규칙

* Markdown 읽기 화면에서는 우측 상단에 문서 액션 아이콘 영역을 둔다.
* 기본은 `연필 아이콘` 으로 표시한다.
* 편집 모드에서는 같은 위치를 `체크 아이콘` 으로 바꾼다.

### 5.2 화면 A: 폴더 목록 화면

현재 경로가 폴더일 때 보여주는 화면이다.

표시 정보

* 상단 breadcrumb
* 현재 폴더명
* 현재 폴더의 하위 폴더 목록
* 현재 폴더의 하위 `.md` 파일 목록

동작

* 폴더를 누르면 해당 폴더로 이동
* Markdown 파일을 누르면 읽기 화면으로 이동
* 폴더 항목은 접기/펼치기 가능

표현 방식

* 폴더는 폴더 카드 또는 폴더 row
* 문서는 문서 카드 또는 문서 row
* 현재 열려 있는 경로는 강조 표시

### 5.3 화면 B: Markdown 읽기 화면

현재 경로가 `.md` 파일일 때 보여주는 화면이다.

표시 정보

* 상단 breadcrumb
* 문서 제목
* 파일명 또는 상대 경로
* 우측 상단 문서 액션 아이콘
* Markdown 본문

동작

* breadcrumb 로 상위 폴더 이동 가능
* 모바일에서 트리 버튼을 눌러 다른 문서로 이동 가능
* 내부 상대 링크는 문서 리더 라우트로 이동
* 우측 상단 `연필 아이콘` 을 누르면 편집 모드로 전환
* 편집 모드에서 우측 상단 `체크 아이콘` 을 누르면 저장 후 다시 읽기 모드로 전환

이 화면에서는 읽기 가독성이 최우선이다.

### 5.4 Markdown 읽기 화면의 편집 모드

편집 모드는 별도 화면이 아니라 **Markdown 읽기 화면 안의 상태 전환**으로 처리한다.

핵심 규칙

* 읽기 모드 우측 상단에는 `연필 아이콘`
* 편집 모드 우측 상단에는 `체크 아이콘`
* 사용자는 현재 보고 있던 문서 자리에서 바로 편집을 시작
* 모드 전환 시 스크롤 위치는 최대한 유지

스크롤 위치 규칙

* 읽기 모드에서 연필 아이콘을 눌러 편집 모드로 들어갈 때 현재 스크롤 위치를 기억
* 편집 모드에서도 가능한 한 같은 위치가 보이도록 에디터를 연다
* 저장 후 읽기 모드로 돌아올 때도 저장 직전의 스크롤 위치 기준으로 다시 보여준다

즉, 모드 전환 때문에 사용자가 다시 위에서부터 문서를 찾지 않게 해야 한다.

편집 완료 규칙

* 편집이 끝나면 우측 상단의 `체크 아이콘` 을 누른다
* 저장 성공 시 즉시 읽기 모드로 전환한다
* 전환 후에는 저장된 최신 Markdown 렌더링 결과를 같은 자리에서 보여준다

## 6. 정보 구조

### 6.1 경로 기반 구조

문서 리더는 별도 DB가 아니라 경로 자체를 상태로 사용한다.

예시

* `/` 또는 `/docs` → 루트 목록
* `/docs/product` → `product/` 폴더 목록
* `/docs/projects/auth-service/deployment.md` → 해당 문서 본문

규칙

* 경로가 폴더면 목록 화면
* 경로가 `.md` 파일이면 문서 화면

### 6.2 README 처리

각 폴더 안에 `README.md` 가 있을 수 있다.

권장 규칙

* 폴더 목록은 항상 보여준다.
* 같은 폴더에 `README.md` 가 있으면 상단에 요약 문서처럼 같이 노출할 수 있다.
* 하지만 폴더를 눌렀을 때 자동으로 `README.md` 로 강제 이동하지는 않는다.

이유

* 사용자는 먼저 “이 폴더에 뭐가 있나”를 보고 싶을 수 있다.
* 폴더 목록과 대표 문서를 동시에 보여주는 편이 더 직관적이다.

## 7. UI 구조

### 7.1 기본 구조

구조는 아래처럼 단순하게 간다.

* 상단: breadcrumb + 현재 위치 + 모바일용 메뉴 버튼 + 문서 액션 영역
* 보조 탐색: 트리 메뉴
* 본문:
  * 폴더면 목록 화면
  * 파일이면 Markdown 읽기 화면
    * 읽기 모드
    * 편집 모드

즉, 기본은 “breadcrumb + 현재 화면 1개” 구조다.

### 7.2 트리 메뉴 규칙

트리 메뉴는 보조 탐색 수단이다.

규칙

* 폴더는 접기/펼치기 가능
* 현재 경로의 상위 폴더는 자동 확장
* 폴더명 클릭은 이동
* 펼침 아이콘 클릭은 열고 닫기만 수행

### 7.3 데스크톱

* 좌측 트리 패널 표시
* 우측에 폴더 목록 화면 또는 Markdown 읽기 화면 표시

### 7.4 모바일

* 좌측 고정 패널은 사용하지 않는다.
* 상단의 트리 버튼을 누르면 드로어 또는 바텀시트로 트리를 연다.
* 기본 화면은 항상 콘텐츠가 전체 폭을 최대한 쓰도록 한다.

## 8. 모바일 친화성 기준

모바일 친화성은 이번 기능의 필수 조건이다.

### 8.1 breadcrumb 동작

모바일에서는 breadcrumb 가 가장 중요하다.

기본 규칙

* 상단에 breadcrumb 를 항상 노출
* 공간이 충분하면 전체 경로 표시
* 공간이 부족하면 **parent folder 까지만 우선 노출**
* 더 앞쪽 경로는 `...` 으로 축약

예시

* 전체: `docs / projects / auth-service / deployment.md`
* 축약: `... / auth-service / deployment.md`
* 폴더 화면 축약: `... / auth-service`

상호작용 규칙

* 앞의 `...` 는 클릭 가능
* `...` 를 누르면 숨겨진 상위 경로 목록이 dropdown 으로 열림
* dropdown 에서 원하는 상위 폴더를 바로 선택 가능

즉, 모바일 breadcrumb 는 단순 말줄임이 아니라 **축약 + 드롭다운 탐색** 구조여야 한다.

### 8.2 탐색

* 리스트 항목 전체를 탭 영역으로 사용
* 최소 터치 영역 44px 이상
* 메뉴는 화면을 밀어내는 고정 사이드바가 아니라 오버레이 드로어 사용
* 폴더는 펼침 상태를 직관적으로 인지할 수 있어야 함

### 8.3 읽기

* 본문 기본 글자 크기 15~16px
* 줄간격 넉넉하게 유지
* 코드 블록은 가로 스크롤 가능
* 표는 가로 스크롤 가능
* 긴 제목과 긴 파일명은 적절히 줄바꿈 또는 말줄임 처리

### 8.4 편집

* 편집 아이콘은 상단 우측에 고정된 위치로 둔다
* 읽기 모드와 편집 모드 전환 시 레이아웃 점프가 크지 않아야 한다
* 모바일에서도 키보드가 열렸을 때 현재 편집 위치를 잃지 않아야 한다
* 저장 후 다시 읽기 모드로 전환될 때 현재 문맥을 유지해야 한다

### 8.5 성능

* 트리 전체를 무겁게 렌더하지 않도록 한다.
* 초기 진입은 현재 경로 기준의 필요한 데이터만 먼저 표시한다.
* 문서 본문 렌더링은 모바일에서 버벅이지 않아야 한다.

## 9. `ingradient-ui` 기반 구현 원칙

이번 UI는 `ingradient-docs` 안에서 직접 스타일을 새로 정의하는 방식보다, **`ingradient-ui`에서 구현하고 재사용 가능한 문서 리더 패턴으로 만드는 방향**이 맞다.

이유

* 문서 리더도 결국 재사용 가능한 탐색/읽기 패턴이다.
* 모바일 드로어, 패널, 카드, breadcrumb, navigation 패턴을 공통화할 수 있다.
* 이후 다른 제품 문서나 내부 가이드에도 같은 UI를 적용하기 쉽다.

### 9.1 구현 위치 권장안

`ingradient-docs` 는 현재 콘텐츠 저장소에 가깝고, 실행 가능한 앱 구조가 없다.

따라서 실제 UI 구현은 아래 중 하나가 적절하다.

* `ingradient-ui/apps/design-showcase` 안에 문서 리더 데모/실페이지 추가
* 또는 `ingradient-ui` 안에 별도 docs-reader app 추가

1차 권장안

* **`ingradient-ui/apps/design-showcase` 에 문서 리더 페이지를 추가**

이유

* 이미 `npm run dev` 흐름이 있다.
* Vite 개발 서버가 준비돼 있다.
* 컴포넌트와 패턴 검증이 빠르다.

### 9.2 재사용 우선 대상

`@ingradient/ui` 에서 우선 활용할 범위

* `patterns` 계열 레이아웃 쉘
* `components` 계열 `Paper`, `Card`, `Accordion`
* 공통 navigation / overlay 계열 패턴
* 브랜드, 토큰, 타이포, surface 스타일

원칙

* 문서 리더 전용 스타일을 최소로 만들고, 가능한 한 `ingradient-ui` 패턴 안에서 해결
* 새로운 UI가 필요하면 `ingradient-ui`에 일반화 가능한 컴포넌트로 추가

## 10. 데이터 공급 방식

브라우저는 로컬 파일 시스템을 직접 자유롭게 읽을 수 없으므로, 문서 트리와 Markdown 본문을 공급하는 계층이 필요하다.

### 10.1 권장 방식: 서버 또는 로컬 API에서 루트 스캔

권장 방식은 아래다.

* 지정된 콘텐츠 루트를 스캔
* 디렉터리 트리를 JSON 으로 반환
* 선택한 `.md` 파일의 raw 내용을 반환

필요 응답 예시

* `GET /api/docs/tree`
* `GET /api/docs/node?path=projects/auth-service`
* `GET /api/docs/content?path=projects/auth-service/deployment.md`
* `PUT /api/docs/content?path=projects/auth-service/deployment.md`

### 10.2 왜 이 방식이 맞는가

* 메뉴 하드코딩이 필요 없다.
* 새 문서 추가 시 자동 반영된다.
* 상대 경로 검증과 루트 탈출 방지를 서버에서 강제할 수 있다.
* 숨김 파일/비문서 파일 필터링 정책을 한곳에서 관리할 수 있다.

### 10.3 경로 보안 규칙

* 요청 경로는 항상 콘텐츠 루트 내부로만 해석
* `..` 로 상위 디렉터리 접근 금지
* `.md` 이외 파일 본문은 기본 비허용
* 쓰기 요청도 `.md` 파일에만 허용

### 10.4 저장 규칙

문서 읽기만이 아니라 인라인 편집을 지원하므로 저장 규칙이 필요하다.

기본 규칙

* 편집 대상은 현재 열려 있는 `.md` 파일 1개
* `체크 아이콘` 클릭 시 현재 편집 내용을 저장
* 저장 성공 후 읽기 모드로 즉시 전환
* 저장 실패 시 편집 모드는 유지하고 오류를 표시

초기 지원 범위

* 리스트
* 링크
* 표
* 코드 블록
* 인용문
* 구분선
* 이미지

링크 정책

* 같은 루트 내 상대 링크는 문서 리더 내부 라우트로 변환

## 12. `npm run dev` 외부 접속 요구사항

문서 리더 개발은 외부 기기 테스트가 가능해야 한다.

필수 조건

* `npm run dev` 실행 시 `0.0.0.0` 에 바인딩
* 고정 포트 사용
* 같은 네트워크의 모바일 기기에서 접속 가능

권장 설정

* Vite `server.host = '0.0.0.0'`
* Vite `preview.host = '0.0.0.0'`
* `strictPort = true`

참고

* 현재 `ingradient-ui/apps/design-showcase/vite.config.ts` 에는 이미 `host: '0.0.0.0'` 설정이 들어가 있다.
* 따라서 문서 리더를 이 앱 안에서 개발하면 외부 접속 요구사항과 잘 맞는다.

## 15. 최종 제안

최종 방향은 아래와 같다.

* `ingradient-docs/docs` 를 콘텐츠 루트로 잡는다.
* 이 루트를 자동 스캔해서 폴더와 `.md` 파일을 메뉴 데이터로 사용한다.
* 화면은 두 개만 둔다: `폴더 목록 화면`, `Markdown 읽기 화면`.
* 두 화면 모두 상단에 breadcrumb 를 둔다.
* 모바일에서 breadcrumb 공간이 부족하면 `... / parent folder / current` 형태로 축약하고, `...` 클릭 시 상위 경로 dropdown 을 연다.
* 폴더는 접기/펼치기 가능한 구조로 탐색한다.
* 폴더를 누르면 목록이 보이고, `.md` 파일을 누르면 Markdown 본문을 보여준다.
* Markdown 읽기 화면 우측 상단에는 `연필 아이콘` 을 두고, 누르면 같은 자리에서 편집 모드로 전환한다.
* 편집 모드에서는 같은 위치를 `체크 아이콘` 으로 바꾸고, 저장 후 다시 읽기 모드로 전환한다.
* 읽기 모드와 편집 모드 전환 전후로 스크롤 위치는 최대한 유지한다.
* UI는 `ingradient-ui` 안에서 구현하고, 모바일 우선으로 설계한다.
* 개발 서버는 `npm run dev` 로 띄웠을 때 `0.0.0.0` 에 바인딩되어 외부 모바일 기기에서도 바로 확인 가능해야 한다.

즉, 이번 문서 리더는 “문서 메뉴를 만드는 작업”이 아니라, **문서 루트를 읽어서 그대로 탐색 가능한 UI로 보여주는 작업**으로 정의하는 것이 맞다.
