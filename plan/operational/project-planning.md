# 프로젝트 계획

```
적용 범위: 운영 환경 전용
강도: Should/Nice 표시
참고: solo / small team / production team 에 따라 적용 수준을 조절한다.
```

> **혼자 혹은 소규모로 빠르게 움직이는 프로젝트에는 가볍게 적용한다.**
> Phase 단위 계획과 완료 기준만 있어도 충분히 효과적이다.

---

## 계획 수준 선택

| 상황 | 권장 |
|------|------|
| 혼자, 프로토타입 | Phase + DoD만 |
| 2-4인 팀 | Phase + DoD + 위험 목록 |
| 5인+ / 외부 의존성 있음 | 전체 템플릿 |

---

## Phase 기반 계획 (Should)

큰 기능이나 리팩터링은 1-3일 단위 Phase로 나눈다.

```markdown
## Phase 1: 데이터 모델 (1일)
- [ ] DB 스키마 설계
- [ ] 마이그레이션 파일 작성
- DoD: `npm test` 통과, DB 쿼리 동작 확인

## Phase 2: 서비스 레이어 (1일)
- [ ] service.ts 작성 + 단위 테스트
- DoD: service 테스트 통과

## Phase 3: UI 연결 (1일)
- [ ] IPC 핸들러 + React 컴포넌트
- DoD: 실제 Electron에서 동작 확인
```

**Should** — Phase가 끝날 때마다 `git commit` (rollback 포인트 확보).

---

## 완료 기준(DoD) 템플릿

```
[ ] 구현 완료
[ ] 단위/통합 테스트 통과
[ ] npm run lint + typecheck 0 errors
[ ] 직접 수동 테스트 완료
[ ] PR 설명에 변경 이유 기술
```

---

## 반드시 멈추고 물어볼 상황 (Should)

계획을 그대로 따르다가 아래 상황이 오면 진행을 멈추고 재논의한다.

- 예상보다 2배 이상 복잡해진 경우
- 다른 기능/팀에 큰 영향을 주는 변경이 필요한 경우
- 보안/데이터 손실 위험이 생긴 경우
- 요구사항이 구현 중간에 바뀐 경우

---

## 위험 관리 (Nice, 팀 프로젝트)

```markdown
| 위험 | 발생 가능성 | 영향 | 대응 |
|------|-----------|------|------|
| 외부 API 변경 | 중 | 높음 | 버전 고정 + 테스트 |
| 마이그레이션 실패 | 낮 | 높음 | 배포 전 스테이징에서 검증 |
```

---

## 체크리스트

```
[ ] 큰 기능을 Phase 단위로 나눴는가?
[ ] 각 Phase에 완료 기준(DoD)이 있는가?
[ ] Phase 완료 시 git commit을 했는가? (롤백 포인트)
[ ] 범위를 벗어나는 변경이 생기면 멈추고 재논의했는가?
```
